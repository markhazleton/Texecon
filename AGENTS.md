# AGENTS.md

## Project Overview
- TexEcon is a static-first React + TypeScript site built with Vite and deployed to GitHub Pages (`texecon.com`).
- Content is sourced from the WebSpark API at build time, cached into `client/src/data`, then used by the SPA and static page generation scripts.
- SEO is a first-class concern: canonical URLs, JSON-LD, sitemap/robots generation, static HTML route generation, and Lighthouse CI checks are part of the repo workflow.

## Project Root

- Vite app root: `client/`
- Build output: `target/` (not `dist/`, unless `TARGET_DIR` is overridden)

## Repository Layout
- `client/index.html`: base HTML template with core SEO/meta tags, crawler links, GA, favicon links.
- `client/public/`: static public assets (`404.html`, `CNAME`, favicons, `site.webmanifest`, generated `sitemap.xml` + `robots.txt`).
- `client/src/components/`: UI and feature components (navigation, SEO, structured data, footer, etc.).
- `client/src/pages/home.tsx`: primary route/content orchestration and dynamic SEO selection.
- `client/src/lib/`: menu, SEO, analytics, utility logic.
- `client/src/data/`: build-generated and cached content data (`webspark-raw.json`, `texecon-content.json`, reports, type file).
- `scripts/`: build pipeline scripts (`clean.js`, `fetch-content.js`, `generate-sitemap.js`, `generate-static-pages.js`, `refresh-content.js`).
- `.github/workflows/deploy.yml`: CI + Pages deploy workflow.
- `.github/lighthouse/lighthouserc.json`: Lighthouse assertions and URLs.

## How The Site Works
1. Build fetches WebSpark content and writes cache/data files.
2. Sitemap/robots are regenerated from `client/src/data/webspark-raw.json`.
3. Vite builds SPA from `client/` into `target/`.
4. Static route HTML files are generated into `target/<api-url>/index.html` using exact API URLs.
5. GitHub Pages serves static output; SPA routing fallback uses `client/public/404.html` and script in `client/index.html`.

## Content Workflow
- Source of truth for dynamic site content: WebSpark API payload cached in `client/src/data/webspark-raw.json`.
- Processed UI data is written to `client/src/data/texecon-content.json`.
- Auto-generated type file: `client/src/data/content-types.ts` (generated by script).
- Team display data also uses `client/src/data/team-data.json`.
- When changing content processing behavior, update scripts first, then verify consuming code in `client/src/lib/data.ts` and `client/src/lib/menu-utils.ts`.

## Navigation, URL, and Content Patterns
- Navigation and hierarchy are derived from API menu data (`display_navigation`, `order`, `parent_page`, `url`, `argument`).
- Canonical path source is `item.url` when present; do not invent alternate URL schemes if API URL exists.
- Current route handling supports `/`, `/:slug`, `/:category/:slug`, plus legacy patterns (`/page/:id`, `/content/:slug`, `/topic/:id`, `/section/:slug`).
- Keep footer, nav menu, sitemap generation, and static page generation aligned to the same URL source (`item.url`).

## Metadata and SEO Expectations
- Preserve base SEO tags in `client/index.html`.
- Dynamic SEO updates are handled in `client/src/components/seo-head.tsx` and `client/src/pages/home.tsx`.
- Structured data is rendered by `client/src/components/structured-data.tsx`; static generated pages also inject JSON-LD in `scripts/generate-static-pages.js`.
- Canonical URLs currently target `https://texecon.com` for indexed pages.
- `scripts/generate-sitemap.js` writes `client/public/sitemap.xml` and `client/public/robots.txt`; build must copy these to `target/`.
- No RSS generator is present in the current codebase. Do not add RSS workflows unless explicitly requested.

## Styling and Component Conventions
- Use existing shadcn/Radix-style component patterns in `client/src/components/ui`.
- Reuse design tokens and utility classes defined in `client/src/index.css` (`--background`, `--foreground`, etc.).
- Tailwind v4 config is CSS-first; `tailwind.config.ts` is intentionally a stub.
- Keep `@/*` alias imports consistent with existing code.
- Maintain existing `data-testid` usage patterns for testability.

## CI/CD and Publishing Expectations
- GitHub Actions (`.github/workflows/deploy.yml`) runs on PRs and pushes to `main`.
- CI checks include: lint, format check, type-check, tests, coverage (non-blocking), build, artifact validation.
- Required artifact expectations include: `target/index.html`, `target/version.json`, `target/sitemap.xml`, `target/robots.txt`.
- Non-PR builds upload Pages artifact; main branch deploy job publishes to GitHub Pages.
- `CUSTOM_DOMAIN` is set to `texecon.com`; workflow may write/update `client/public/CNAME`.

## Approved Patterns To Follow
- Prefer extending existing scripts/components over introducing parallel systems.
- Keep changes consistent with existing API-driven architecture (menu hierarchy, SEO path generation).
- Use strict TypeScript-compatible patterns; avoid introducing `any` unless already unavoidable.
- Add or update tests near changed behavior (`*.test.ts` / `*.test.tsx`).
- Preserve static generation and crawler discoverability behavior.

## Guardrails (Minimize Drift)
- Do not add new dependencies unless required and justified by an existing gap.
- Do not replace routing, build, or styling frameworks without explicit approval.
- Do not hardcode new content structures when API fields already exist.
- Do not manually edit generated artifacts in `target/`; regenerate via scripts/build.
- Avoid broad refactors touching URL/SEO logic without validating sitemap + static page output.

## Areas Requiring Extra Caution
- `scripts/fetch-content.js`, `scripts/generate-sitemap.js`, `scripts/generate-static-pages.js`: these define publishing/indexing behavior.
- `client/index.html`: central SEO/canonical/social metadata baseline.
- `client/src/pages/home.tsx` and `client/src/lib/seo-utils.ts`: route resolution and canonical/meta behavior.
- `.github/workflows/deploy.yml`: production publishing path.
